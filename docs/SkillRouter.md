| Step | E-shop algo | SkillRouter algo | Same? |
|---|---|---|---|
| **Overall structure** | Fast retrieval model → slow reranker | Fast retrieval model → slow reranker | ✅ Same |
| **What you index** | Name + short description | Name + description + full body text | ❌ Different |
| **Retrieval model type** | Bi-encoder (embed query, embed item, compare vectors) | Bi-encoder (embed query, embed item, compare vectors) | ✅ Same |
| **How you build training examples** | Right answer = positive, everything else = negative, done | Right answer = positive, then check each negative — if it does the same thing as the right answer, remove it from training | ❌ Different |
| **Retrieval training loss** | InfoNCE contrastive loss | InfoNCE contrastive loss | ✅ Same |
| **How retrieval works at inference** | Embed query → find top 20 closest items by vector similarity | Embed query → find top 20 closest items by vector similarity | ✅ Same |
| **What you feed the reranker** | Query + item name + description | Query + item name + description + full body | ❌ Different |
| **Reranker model type** | Cross-encoder (reads query and item together) | Cross-encoder (reads query and item together) | ✅ Same |
| **How you train the reranker** | Score each item independently: "relevant? yes/no" (pointwise) | Show the whole candidate list, teach the model which item beats which (listwise) | ❌ Different |
| **How reranking works at inference** | Score each item → sort by score | Score each item → sort by score | ✅ Same |
| **Extra signals** | Clicks, purchases, ratings, price | None — text only | ❌ Different |

**Bottom line:** the skeleton is identical. The differences are three targeted fixes to three specific problems that only exist in homogeneous pools: what text you feed in, one extra filtering step in training data preparation, and a different training objective for the reranker.

Based on what the paper describes, here is an honest breakdown.

---

## What they actually compared against

They ran their system against these baselines:

- BM25 (classic keyword search)
- Qwen3-Emb-8B — a large general-purpose embedding model, zero-shot (not fine-tuned for skills)
- Qwen3-Rank-8B — a large general-purpose reranker, zero-shot
- Qwen3-Emb-8B + Qwen3-Rank-8B combined (16B total parameters)

Their 1.2B fine-tuned system beat the 16B zero-shot system by 6 points on Hit@1, at 5.8x faster latency.

So they did run real comparisons, not just self-reported results.

---

## The benchmark — who made it?

**They made it themselves.** This is the important caveat.

- 75 expert-verified queries, built by the paper's authors
- ~80K skill pool assembled from SkillsBench and open-source collections, also by the authors
- Hard distractors generated by them using an LLM
- Graded relevance labels assigned by them

This is normal in ML papers, but it means they both designed the test and the solution. The benchmark was designed specifically to capture the problem SkillRouter is designed to solve — large homogeneous pools with overlapping skills. A benchmark designed differently might tell a different story.

---

## Did they test on a real agent?

Yes, they ran a downstream end-to-end test with four actual coding agents:

- Claude Sonnet
- Claude Opus
- Kimi-K2.5
- GLM-5

On 75 tasks from their benchmark, the task success rate went from 25.78% (baseline router) to 27.56% (SkillRouter). The gap was bigger for stronger agents (Claude models +3.22pp, smaller models +0.89pp).

These are real agents, real tasks, real end-to-end runs — not simulated.

---

## Where the evaluation is genuinely strong

- Baselines are real systems, not strawmen — a 16B zero-shot pipeline is a serious baseline
- They open-sourced the benchmark, the evaluation code, and the models — anyone can reproduce it
- End-to-end downstream agent test validates the retrieval improvement actually helps task completion
- Two difficulty tiers (easy/hard) show robustness, not just cherry-picked results

---

## Where the evaluation has limitations

**Baselines are all zero-shot, their model is fine-tuned.** This is the biggest issue. They compare a fine-tuned 1.2B model against a zero-shot 16B model. The fair comparison would be: fine-tune the 16B model the same way and see if their specific training recipe still wins, or if it was just fine-tuning that mattered. They do not run this comparison.

**They own the benchmark.** The benchmark was built to illustrate exactly the problem their method solves. There is no independent third-party evaluation on a pre-existing benchmark that their method was not designed around.

**No competitor specifically for skill routing.** There is no prior system that specifically solves large-scale skill routing with fine-tuning. They compare against general retrieval models, not against another team's solution to the same problem. This is because the problem definition itself is new — but it means "better than previous work" is partly true by construction.

---

## Short verdict

The comparison is real and honest in what it tests. The downstream agent experiments are genuine. But the evaluation is self-contained: they defined the problem, built the benchmark, designed the solution, and reported results. The numbers are reproducible (open code and data), but no independent lab has validated the claims on a neutral benchmark yet. That is standard for a new paper, not a red flag — but it is the correct way to read the confidence level of the results.
